TDE03 de Back-end
Documentacao Tecnica do Sistema

====================================================================
1) O Que Foi Desenvolvido
====================================================================

1.1 Autenticacao e sessao
- Tela de login (`login.php`) para entrada no sistema.
- Tela de registro (`registro.php`) para criacao de novos usuarios.
- Validacao de credenciais usando tabela `usuarios`.
- Controle de sessao para impedir acesso nao autenticado ao painel.
- Tela de logout (`logout.php`) para encerramento da sessao.

1.2 Roteamento centralizado
- Todas as acoes principais passam por `index.php`.
- Rota baseada em dois parametros:
  - `entidade` (modulo)
  - `acao` (operacao)
- Operacoes padrao implementadas por entidade:
  - listar
  - novo
  - editar
  - salvar
  - excluir

1.3 Modulos implementados
- Proprietarios.
- Corretores.
- Clientes.
- Imoveis.
- Visitas.
- Contratos.

1.4 Regras e recursos relevantes
- Upload de planta baixa no modulo de imoveis.
- Bloqueio de novo contrato para imovel indisponivel.
- Atualizacao automatica do status do imovel apos salvar contrato.
- Calculo automatico de periodo da visita (manha/tarde/noite)
  com base no horario informado.

1.5 Persistencia e integridade de dados
- Persistencia com PDO e SQLite (`database.sqlite`).
- Uso de chaves estrangeiras para garantir relacionamentos validos.
- Uso de restricoes `CHECK` no schema para valores de dominio.

====================================================================
2) Telas Desenvolvidas
====================================================================

2.1 Tela de login
- Arquivo: `login.php`
- Objetivo: autenticar usuario para liberar acesso ao sistema.
- Entradas: e-mail e senha.
- Saida esperada: sessao ativa e redirecionamento para `index.php`.

2.2 Tela de registro de usuario
- Arquivo: `registro.php`
- Objetivo: cadastrar novo usuario na tabela `usuarios`.
- Entradas: nome, e-mail, senha, confirmar senha.
- Validacoes implementadas:
  - campos obrigatorios
  - formato valido de e-mail
  - senha igual a confirmacao
  - e-mail nao duplicado
- Saida esperada: mensagem de sucesso e orientacao para login.

2.3 Tela principal / painel
- Arquivo: `index.php` (estado home, quando `entidade=home`).
- Objetivo: ser a pagina central de navegacao do sistema.
- Conteudo:
  - menu de modulos
  - atalhos de acao
  - indicadores gerais (totais por entidade)

2.4 Formulario de proprietario
- Arquivo: `view/proprietario/form.php`
- Campos: nome, cpf, telefone, email.
- Objetivo: criar/editar proprietarios.
- Observacao: em edicao, envia `id` oculto para atualizar registro.

2.5 Formulario de corretor
- Arquivo: `view/corretor/form.php`
- Campos: nome, creci, telefone, email.
- Objetivo: criar/editar corretores.
- Observacao: `creci` e campo obrigatorio por regra de negocio.

2.6 Formulario de cliente
- Arquivo: `view/cliente/form.php`
- Campos: nome, cpf, telefone, email, interesse.
- Objetivo: criar/editar clientes.
- Observacao: `interesse` define se cliente busca compra ou aluguel.

2.7 Formulario de imovel
- Arquivo: `view/imovel/form.php`
- Campos:
  - titulo
  - tipo
  - endereco
  - metros_quadrados
  - valor
  - finalidade
  - status
  - planta_baixa
  - id_proprietario
- Objetivo: criar/editar imoveis.
- Regras importantes:
  - upload de planta baixa com `multipart/form-data`
  - preservacao da planta atual em edicao quando nao ha novo upload

2.8 Formulario de visita
- Arquivo: `view/visita/form.php`
- Campos: id_imovel, nome, email, celular, dia_semana, horario_preferencia.
- Objetivo: agendar/editar visitas.
- Regra importante: periodo e derivado do horario no controller.

2.9 Formulario de contrato
- Arquivo: `view/contrato/form.php`
- Campos: id_imovel, id_cliente, id_corretor, tipo, valor, data_inicio, data_fim.
- Objetivo: criar/editar contratos.
- Regras importantes:
  - cruza 3 entidades (imovel + cliente + corretor)
  - imoveis indisponiveis aparecem bloqueados para novo contrato

2.10 Listagens por modulo
- Renderizadas no `index.php` conforme `entidade` ativa.
- Cada listagem permite:
  - abrir formulario de novo registro
  - editar registro existente
  - excluir registro

3) Arquitetura e Fluxo Tecnico
====================================================================

3.1 Diagrama de componentes

  [Navegador]
      |
      v
  [login.php / registro.php]
      |
      +--> [index.php / Front Controller]
                 |
                 v
          [Controller da Entidade]
                 |
                 v
              [Model]
                 |
                 v
              [DAO] ---- PDO ----> [SQLite]
                 |
                 v
               [View]

3.2 Como o fluxo funciona na pratica
1. O usuario interage com uma tela (formulario ou listagem).
2. A requisicao chega ao `index.php` com `entidade` e `acao`.
3. O `index.php` direciona para o controller correto.
4. O controller organiza e valida os dados da entrada.
5. O controller monta/atualiza um objeto Model.
6. O DAO persiste ou consulta os dados no banco.
7. O resultado volta para a view de listagem ou formulario.

3.3 Por que esse formato foi usado
- Separacao de responsabilidades facilita manutencao.
- Reduz acoplamento entre tela e SQL.
- Permite estudar cada camada isoladamente.

4) Controllers (Explicacao Completa)
====================================================================

Conceito geral
- Controller e a camada de orquestracao da requisicao.
- Ele recebe dados brutos da entrada e transforma para o dominio.
- Nao deve conter SQL direto (isso fica no DAO).

4.1 `ImovelController` (`controller/ImovelController.php`)
- Funcoes principais:
  - `listar()`
  - `buscarPorId()`
  - `listarProprietarios()`
  - `salvar()`
  - `excluir()`
- Responsabilidades especificas:
  - mapear dados do formulario para objeto `Imovel`
  - tratar upload da planta baixa
  - manter planta antiga quando nao ha novo envio

4.2 `ProprietarioController` (`controller/ProprietarioController.php`)
- Funcoes principais: listar, buscarPorId, salvar, excluir.
- Responsabilidade: gerenciar ciclo de vida de proprietarios.

4.3 `CorretorController` (`controller/CorretorController.php`)
- Funcoes principais: listar, buscarPorId, salvar, excluir.
- Responsabilidade: gerenciar dados profissionais dos corretores.

4.4 `ClienteController` (`controller/ClienteController.php`)
- Funcoes principais: listar, buscarPorId, salvar, excluir.
- Responsabilidade: gerenciar clientes e interesse comercial.

4.5 `VisitaController` (`controller/VisitaController.php`)
- Funcoes principais:
  - listar, buscarPorId, salvar, excluir
  - listarImoveis
- Responsabilidade adicional:
  - calcular `periodo` pela funcao `periodoPorHorario()`.

4.6 `ContratoController` (`controller/ContratoController.php`)
- Funcoes principais:
  - listar, buscarPorId, salvar, excluir
  - listarImoveis, listarClientes, listarCorretores
- Regras de negocio importantes:
  - impede novo contrato para imovel nao disponivel
  - apos salvar contrato, atualiza status do imovel para:
    - `alugado` (tipo aluguel)
    - `vendido` (tipo venda)

5) DAOs (Explicacao Completa)
====================================================================

Conceito geral
- DAO (Data Access Object) e a camada que fala com o banco.
- Centraliza SQL, parametrizacao e mapeamento de resultado.

Arquivos
- `dao/ImovelDAO.php`
- `dao/ProprietarioDAO.php`
- `dao/CorretorDAO.php`
- `dao/ClienteDAO.php`
- `dao/VisitaDAO.php`
- `dao/ContratoDAO.php`

Metodos comuns
- `listar()`: consulta colecao de registros.
- `buscarPorId($id)`: consulta unico registro por ID.
- `salvar($objeto)`: INSERT (novo) ou UPDATE (edicao).
- `excluir($id)`: DELETE por ID.

Boas praticas aplicadas
- Uso de `prepare/execute` para consultas parametrizadas.
- Conversao de linhas SQL em objetos Model.
- Isolamento da persistencia fora do controller.

====================================================================
6) Models (Explicacao Completa)
====================================================================

Conceito geral
- Model representa entidade de negocio no codigo.
- Armazena estado e expone acesso por getters/setters.

Arquivos
- `model/Imovel.php`
- `model/Proprietario.php`
- `model/Corretor.php`
- `model/Cliente.php`
- `model/Visita.php`
- `model/Contrato.php`

Papel pratico
- Padronizar os campos de cada entidade.
- Facilitar validacao e manipulacao de dados entre camadas.
- Evitar passar arrays soltos no sistema inteiro.

Exemplo de uso no fluxo
- Controller recebe `$_POST`.
- Preenche objeto Model com tipos corretos.
- Envia objeto para DAO persistir.

7) Conexao e Banco de Dados
====================================================================

7.1 Classe de conexao
- Arquivo: `config/conexao.php`
- Implementacao: Singleton via `Conexao::getConn()`.
- Banco atual: SQLite em arquivo local `database.sqlite`.
- Configuracoes:
  - `PDO::ATTR_ERRMODE = PDO::ERRMODE_EXCEPTION`
  - `PRAGMA foreign_keys = ON`

7.2 Schema
- Arquivo: `config/schema_sqlite.sql`
- Tabelas existentes:
  - usuarios
  - proprietarios
  - corretores
  - clientes
  - imoveis
  - visitas
  - contratos

7.3 Relacionamentos principais
- `imoveis.id_proprietario -> proprietarios.id`
- `visitas.id_imovel -> imoveis.id`
- `contratos.id_imovel -> imoveis.id`
- `contratos.id_cliente -> clientes.id`
- `contratos.id_corretor -> corretores.id`

7.4 Regras de dominio no banco
- Campos unicos para evitar duplicidade (ex.: cpf, creci, email).
- `CHECK` para valores controlados (ex.: tipo, status, interesse).
- Foreign keys para garantir consistencia relacional.

8) Fluxos Principais de Uso
====================================================================

8.1 Registro de usuario
1. Usuario acessa `registro.php`.
2. Preenche nome, e-mail, senha e confirmacao.
3. Sistema valida campos e formato.
4. Sistema verifica duplicidade por e-mail.
5. Se valido, cria usuario em `usuarios`.
6. Usuario segue para login.

8.2 Login
1. Usuario acessa `login.php`.
2. Informa e-mail e senha.
3. Sistema valida no banco.
4. Sessao e criada.
5. Redirecionamento para `index.php`.

8.3 Cadastro de imovel
1. Acessar `index.php?entidade=imovel&acao=novo`.
2. Preencher formulario, opcionalmente com planta baixa.
3. Controller monta objeto `Imovel`.
4. DAO persiste em `imoveis`.
5. Listagem e atualizada.

8.4 Agendamento de visita
1. Acessar `index.php?entidade=visita&acao=novo`.
2. Escolher imovel e informar dados do visitante.
3. Controller calcula periodo pelo horario.
4. DAO grava registro em `visitas`.

8.5 Fechamento de contrato
1. Acessar `index.php?entidade=contrato&acao=novo`.
2. Selecionar imovel, cliente e corretor.
3. Controller valida disponibilidade do imovel.
4. DAO grava contrato em `contratos`.
5. Sistema atualiza status do imovel.

9) Estrutura de Pastas
====================================================================

`config/`
- Arquivos de infraestrutura e schema do banco.

`controller/`
- Orquestracao dos fluxos e regras de entrada.

`dao/`
- Persistencia e queries SQL.

`model/`
- Entidades e estrutura de dados do dominio.

`view/`
- Telas de formulario e renderizacao das listagens.

`uploads/`
- Arquivos enviados no sistema (planta baixa).

`tests/`
- Script de verificacao automatizada do sistema.

====================================================================
10) Setup e Testes
====================================================================

10.1 Pre-requisitos
- PHP 8+.
- Extensao PDO com SQLite habilitada.

10.2 Execucao local
1. Confirmar existencia de `database.sqlite` na raiz.
2. Iniciar servidor local:
   `php -S localhost:8000`
3. Acessar no navegador:
   - `http://localhost:8000/registro.php`
   - `http://localhost:8000/login.php`

10.3 Testes
- Comando: `php tests/run.php`
- Escopo:
  - conexao com banco
  - validacao de tabelas
  - verificacao de usuario
  - CRUD por modulo

11) Glossario
====================================================================

CRUD
- Create, Read, Update, Delete.

MVC
- Model, View, Controller.

DAO
- Data Access Object (camada de acesso ao banco).

PDO
- API de acesso a banco no PHP.

Front Controller
- Ponto unico de entrada da aplicacao (`index.php`).

Sessao
- Dados temporarios do usuario no servidor.

PRG
- Post/Redirect/Get (evita reenvio de formulario).

PK
- Primary Key (chave primaria).

FK
- Foreign Key (chave estrangeira).

CRECI
- Registro profissional do corretor de imoveis.
